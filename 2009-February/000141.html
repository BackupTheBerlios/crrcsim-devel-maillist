Here are some notes about the subject..maybe this is just my two cents,
maybe we can stick to it... 

 - coding style (we agreed on that some time ago)
     * no tabs
     * indentation, braces
     * no further code on if/else lines despite condition
 - no inactive (commented out) code without good reason and comment. We DO
   have a VCS.
 - code quality in general
 - don't push what's far from finished if it is visible to the user or in
   the config files and you don't manage to fix it for the next release 
 - if you messed up your local repo by adding much-too-big-files, pointless
   merge changesets or something like that: don't push. Ask on the list about
   how to fix your local repo (it is possible!), fix it, push afterwards.   
 - take a look at your changesets regarding the points above before you push 

The current state of CRRCSim is too good to leave traces of bad
coding/style, obvious regressions or something like this. And we have little
manpower -- so if we do something, it should be done right or not done at all.

If something worked (well) for a user in version A, it should work at least
equally well in version B, no matter if it is a fresh install or an upgrade.
We must not require a user to tweak config files in order to get something
working again or fix misfeatures.

Best regards,
Jens
